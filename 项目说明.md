# 项目说明：内容审核规则引擎

小组 101 · 课题：规则引擎与基础策略
方向：不良价值观内容识别 · 目标分：85（方向还未最终确定，但是可以根据敏感词库的情况来看，最好今天定下来，基准分也可以商量）

---

## 一、项目目标

做一个审核内容的规则引擎：给定一段内容，引擎判断它是否违规，并输出处理结果。

```
内容输入  →  规则引擎  →  放行 / 转人工 / 拦截
```

当前聚焦**不良价值观**方向（拜金炫富、低俗审丑、暴力歧视宣扬等）。引擎与产域解耦，以后要加别的方向，只需补规则配置，不用改引擎本身。

---

## 二、判定结果：三态

不做"违规/不违规"的二分类，而是三种结果，给机器拿不准的情况留一个出口：

| 结果 | 含义 | 触发场景 |
|---|---|---|
| 放行 | 通过 | 内容干净 |
| 转人工 | 交人工审核 | 灰色地带，机器不确定 |
| 拦截 | 拦下 | 明确违规 |


---

## 三、引擎怎么工作

引擎核心是一组判断工具，对输入内容逐项检查：

- **词库**：是否命中敏感词（用 AC 自动机，匹配速度快）
- **规则**：是否符合特定模式（正则、特征）
- **打分**：把命中的结果加权算出一个分数（0~1）
- **判定**：分数对照阈值，给出三态结果之一

```
内容
  ↓
词库匹配 + 规则匹配
  ↓
分级打分（0~1）
  ↓
分低 → 放行    分中 → 转人工    分高 → 拦截
```

---

## 四、关键设计

**1. 规则配置化，不硬编码**
规则写成 JSON 配置，新增规则等于改配置，无需改代码重新部署。这是"规则引擎"区别于硬编码 if-else 的核心，也是课题考核重点。

**2. 危害分三级（L1/L2/L3）**
- L3 高危：明确严重
- L2 中危：需结合上下文
- L1 低危：擦边、疑似

分数主要由命中的最高等级决定，保证判定可解释、可追溯。

**3. 处理尺度三档**
同一套规则支持宽松 / 标准 / 严格三档：
- 宽松：只拦最严重的
- 严格：宁可错杀，多拦、多转人工

不良价值观边界模糊，建议默认保留较大的"转人工"区间。

## 六、开源选型

老师建议先调研现成模块，调研结论如下。原则是：**底层工具直接复用，引擎内核自研**（内核是考核核心，不能外包）。

| 模块 | 开源参考 | 用法 |
|---|---|---|
| 词库 | Sensitive-lexicon | 直接用，补充不良价值观词 |
| 词库匹配 | pyahocorasick | 直接用 |
| 引擎内核 | ZEN Engine / json-rules-engine | 参考设计，自研 |
| 可视化后台 | Langflow / GoRules | 借鉴界面，不当引擎 |
| 消息队列 | Redis | 直接用 |

关于 Langflow：它是 AI 工作流的可视化编排工具，适合作可视化参考或后续模型那一环，不适合当引擎内核。是否使用、用在哪，建议与老师确认一次。

## 十、风险提示

- **数据来源**：标注数据获取难度高，最容易拖进度，尽早确认来源、尽早攒词库。
- **判定难度**：不良价值观语义性强、边界主观，规则难覆盖的部分交给模型接口或转人工，不硬扛。
- **范围控制**：图片、语音、模型均先留接口，不实现，核心是规则引擎。
- **合规**：测试数据用脱敏特征与公开评测集，敏感原始数据不存储，必要时与老师确认。
